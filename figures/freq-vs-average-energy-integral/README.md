This figure plots GPU core frequency against two windowed metrics:

- Left y-axis: average completed-agent request energy, computed from a per-request power model.
- Right y-axis: average throughput.

The intended source data is:

- `/srv/scratch/yichaoy2/work/vllm-otel/results/replay/single-qps-sweep-freq-clean/swebench-verified/mini-swe-agent/split/exclude-unranked/qps0_08`

`start` and `end` are seconds from experiment start. For example, `--start-s 5000 --end-s 8000` means "only use data in the 5000s to 8000s window". If `--end-s` is omitted, each run uses its full available analysis window.

This directory follows the top-level `figures/README.md` rule:

1. Step 1 materializes a figure-specific dataset into `data/`.
2. Step 2 renders the figure from that processed dataset only.

**Metric Definition**
Compared with `figures/freq-vs-average-energy/`, this figure uses request-level energy attribution instead of `avg_power * duration / finished_count`.

For each run and selected time window:

1. Build a per-request power model:
   - Read total GPU power from `post-processed/power/power-summary.json`.
   - Read `vllm:num_requests_running` from `post-processed/vllm-log/gauge-counter-timeseries.json`.
   - At each power sample time `t`, define:
     - `per_request_power(t) = total_power(t) / num_requests_running(t)` when `num_requests_running(t) > 0`
     - `per_request_power(t) = 0` when `num_requests_running(t) == 0`
2. Select the finished agents:
   - Use `replay/summary.json` to find workers whose `finished_at` falls inside the selected window.
   - Map those trials to gateway `trace_id`s using `post-processed/global/trial-timing-summary.json`.
3. Attribute request energy to each finished agent:
   - Read request intervals from `post-processed/gateway/llm-requests/llm-requests.json`.
   - For every request generated by a selected finished agent, integrate `per_request_power(t)` over the part of the request that overlaps the selected window.
   - Sum those request energies to get the agent's energy.
4. Average over finished agents:
   - `average_completed_agent_request_energy = mean(agent_energy_j for mapped finished agents in window)`
5. Throughput uses the same windowed completion definition as the simpler figure:
   - `average_throughput = finished_replay_count_in_window / window_duration_s`

This way the left axis reflects request-attributed energy for completed agents only, which is closer to the intent described in the original note.

**Step 1**
Materialize the processed data into `data/`:

```bash
python3 figures/freq-vs-average-energy-integral/materialize_integral_metrics.py \
  --root-dir /srv/scratch/yichaoy2/work/vllm-otel/results/replay/single-qps-sweep-freq-clean/swebench-verified/mini-swe-agent/split/exclude-unranked/qps0_08
```

Example with an explicit window:

```bash
python3 figures/freq-vs-average-energy-integral/materialize_integral_metrics.py \
  --root-dir /srv/scratch/yichaoy2/work/vllm-otel/results/replay/single-qps-sweep-freq-clean/swebench-verified/mini-swe-agent/split/exclude-unranked/qps0_08 \
  --start-s 5000 \
  --end-s 8000
```

By default this writes a gitignored CSV like:

- `figures/freq-vs-average-energy-integral/data/freq-vs-average-energy-integral.start-0.end-full.csv`
- `figures/freq-vs-average-energy-integral/data/freq-vs-average-energy-integral.start-5000.end-8000.csv`

Key columns in the CSV:

- `frequency_mhz`
- `selected_finished_replay_count_in_window`
- `mapped_finished_replay_count_in_window`
- `total_request_integral_energy_across_finished_replays_j`
- `average_request_integral_energy_per_finished_replay_j`
- `average_throughput_jobs_per_s`
- `window_total_power_energy_integral_j`

The CSV also records how many finished agents could be mapped back to request traces. In the target sweep these counts are expected to match, but the materialized data keeps both values explicit.

**Step 2**
Render the dual-axis figure from the materialized CSV:

```bash
python3 figures/freq-vs-average-energy-integral/plot_freq_vs_average_energy_integral.py \
  --input figures/freq-vs-average-energy-integral/data/freq-vs-average-energy-integral.start-0.end-full.csv
```

Example with a custom output path:

```bash
python3 figures/freq-vs-average-energy-integral/plot_freq_vs_average_energy_integral.py \
  --input figures/freq-vs-average-energy-integral/data/freq-vs-average-energy-integral.start-5000.end-8000.csv \
  --output figures/freq-vs-average-energy-integral/output/freq-vs-average-energy-integral.start-5000.end-8000.pdf
```

The default rendered output goes into the gitignored `output/` directory.

The plotting step requires `matplotlib` to be installed in the active Python environment.
